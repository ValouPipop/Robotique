\begin{abstract}
This report presents the design and implementation of an autonomous navigation system for a TurtleBot3 robot in a simulated restaurant. The robot must deliver orders from the kitchen to five tables while avoiding obstacles. Two paradigms are compared: \textbf{classical path planning} (A*, Dijkstra, Greedy + DWA controller) and \textbf{reinforcement learning} (PPO Actor-Critic). The comparison evaluates path efficiency, success rate, obstacle avoidance, and computation time.
\end{abstract}

\section{Introduction}

\subsection{Use Case}

\begin{tcolorbox}[colback=lightblue, colframe=primaryblue, title=\textbf{Use Case: Restaurant Robot Waiter}]
A restaurant wants to automate table service using a mobile robot. The robot must autonomously navigate from the kitchen to \textbf{5 different tables} in the dining area, avoiding walls, furniture, and obstacles along the way. The goal is to compare two approaches: classical algorithms vs.\ artificial intelligence.
\end{tcolorbox}

\subsection{Objectives}
\begin{enumerate}
  \item Build a simulated restaurant environment using Gazebo and ROS
  \item Implement classical path planning (A*, Dijkstra, Greedy) with a DWA controller
  \item Implement a reinforcement learning agent (PPO Actor-Critic)
  \item Compare both paradigms on path efficiency, success rate, obstacle avoidance, and computation time
\end{enumerate}

\subsection{System Overview}
The system uses a \textbf{TurtleBot3 Burger} simulated in Gazebo. The robot starts at position $(19.0, 9.0)$ in the kitchen and must reach one of 5 tables in the dining area.

\section{Environment Setup}

\subsection{Restaurant World}
The restaurant was modeled in Gazebo using an aerial photograph as reference. The \texttt{.world} file contains walls (box geometries), 5 tables (\texttt{cafe\_table} models), and a kitchen area. Table positions are stored in a JSON configuration file.

\subsection{Map Generation}
The occupancy grid map was generated using \textbf{SLAM} (\texttt{slam\_toolbox}). Initial attempts with \texttt{explore\_lite} and \texttt{gmapping} failed due to wheel slippage. A Python script was used to capture the map reliably.

\begin{tcolorbox}[colback=lightorange, colframe=warnorange, title=\textbf{Map Parameters}]
\begin{itemize}
  \item \textbf{Resolution}: $0.05$ m/pixel \quad | \quad \textbf{Origin}: $(-2.0, -2.0)$
  \item \textbf{Occupied threshold}: $0.65$ \quad | \quad \textbf{Free threshold}: $0.196$
  \item \textbf{Format}: PGM (Portable Gray Map) + YAML metadata
\end{itemize}
\end{tcolorbox}

\subsection{Coordinate Transformation}
Gazebo coordinates (meters) are converted to grid coordinates (pixels) using:
\begin{equation}
\text{col}_x = \text{round}\left(\frac{x_m - x_{\text{origin}}}{\text{resolution}}\right), \qquad
\text{row}_y = (H - 1) - \text{round}\left(\frac{y_m - y_{\text{origin}}}{\text{resolution}}\right)
\end{equation}

\subsection{Obstacle Inflation}
To prevent collisions, obstacles are inflated (dilated) on the grid:
\begin{equation}
r_{\text{inflation}} = \left\lceil \frac{r_{\text{robot}} + r_{\text{margin}}}{\text{resolution}} \right\rceil = \left\lceil \frac{0.20 + 0.05}{0.05} \right\rceil = 5 \text{ pixels}
\end{equation}
A circular structuring element is used with SciPy's \texttt{binary\_dilation}.

\section{Classical Navigation}

\subsection{Path Planning Algorithms}

Three graph-search algorithms were implemented on the inflated 4-connected grid.

\subsubsection{A* Algorithm}
A* combines actual cost $g(n)$ with a heuristic $h(n)$:
\begin{equation}
\colorbox{lightblue}{$f(n) = g(n) + h(n)$}, \qquad h(n) = |x_n - x_g| + |y_n - y_g| \text{ (Manhattan)}
\end{equation}
A* is \textbf{optimal} and \textbf{complete} when $h(n)$ is admissible.

\subsubsection{Dijkstra's Algorithm}
Dijkstra is A* with $h(n) = 0$:
\begin{equation}
\colorbox{lightgreen}{$f(n) = g(n)$}
\end{equation}
It guarantees the shortest path but explores more nodes (no directional guidance).

\subsubsection{Greedy Best-First Search}
Greedy uses only the heuristic:
\begin{equation}
\colorbox{lightorange}{$f(n) = h(n)$}
\end{equation}
Fastest algorithm, but \textbf{not optimal} --- the path may be longer than the shortest.

\subsection{DWA Controller (Path Following)}

The \textbf{Dynamic Window Approach} translates the grid path into velocity commands $(v, \omega)$ for the real robot in Gazebo.

\subsubsection{Kinematic Model}
\begin{equation}
\begin{cases}
x_{t+1} = x_t + v \cos(\theta_t) \cdot \Delta t \\
y_{t+1} = y_t + v \sin(\theta_t) \cdot \Delta t \\
\theta_{t+1} = \theta_t + \omega \cdot \Delta t
\end{cases}
\quad \text{with } \Delta t = 0.2 \text{ s}
\end{equation}

\subsubsection{Dynamic Window}
Feasible velocities are constrained by acceleration limits:
\begin{equation}
V_s = \{(v, \omega) \mid v \in [v_c - a_v \Delta t,\ v_c + a_v \Delta t],\ \omega \in [\omega_c - a_\omega \Delta t,\ \omega_c + a_\omega \Delta t]\}
\end{equation}
with $v_{\max} = 0.5$ m/s, $\omega_{\max} = 1.0$ rad/s, $a_v = 0.5$ m/s$^2$, $a_\omega = 1.0$ rad/s$^2$.

\subsubsection{Cost Function}
Each candidate trajectory is evaluated over a horizon of $H = 5$ steps:
\begin{equation}
J(v, \omega) = \underbrace{2.0 \cdot d_{\text{path}}}_{\text{stay on path}} + \underbrace{1.0 \cdot \Delta\theta_{\text{heading}}}_{\text{face the goal}} + \underbrace{0.5 \cdot (v_{\max} - v)}_{\text{go fast}}
\end{equation}

\section{Reinforcement Learning Navigation}

\subsection{Why Actor-Critic?}

\begin{tcolorbox}[colback=lightred, colframe=accentred, title=\textbf{Why PPO Actor-Critic instead of DQN or Q-Learning?}]
The TurtleBot3 is controlled with \textbf{continuous velocity commands}: linear velocity $v \in [0, 0.5]$ m/s and angular velocity $\omega \in [-1.0, 1.0]$ rad/s.

Classical RL methods like \textbf{DQN} (Deep Q-Network) only work with \textbf{discrete actions} (e.g., ``turn left'', ``go forward''). This would force us to discretize the velocity space, losing precision and smooth control.

\textbf{Actor-Critic} methods (like PPO) natively support \textbf{continuous action spaces}:
\begin{itemize}
  \item The \textbf{Actor} outputs a Gaussian distribution $\mathcal{N}(\mu, \sigma^2)$ over velocities
  \item Actions are sampled continuously, bounded by the TurtleBot3's physical limits
  \item This produces \textbf{smooth, natural trajectories} instead of jerky discrete steps
\end{itemize}
\end{tcolorbox}

\subsection{Observation Space (26 values = the ``State'')}

\begin{tcolorbox}[colback=lightblue, colframe=primaryblue, title=\textbf{What the robot ``sees'' at each step (26 numbers)}]
The agent receives a vector of 26 continuous values:

\begin{enumerate}
  \item \textbf{24 LiDAR rays} -- Simulated laser scanner covering 360\textdegree{} around the robot (one ray every 15\textdegree{}). Each ray returns the \textbf{distance to the nearest obstacle} (0 to 3.5\,m). This is how the robot ``sees'' walls and furniture.

  \item \textbf{1 Dijkstra distance} -- The \textbf{true shortest-path distance} to the target table, computed via a pre-calculated distance map (Dijkstra flood-fill). Unlike Euclidean distance, this accounts for walls and corridors.

  \item \textbf{1 Heading angle} -- The angle between the robot's current orientation and the direction to the goal ($\phi \in [-\pi, \pi]$). This tells the robot which way to turn.
\end{enumerate}
\end{tcolorbox}

\subsection{Action Space (2 continuous values)}

The Actor outputs two values, directly mapped to robot commands:
\begin{equation}
a = (v, \omega) \in [0.0, 0.5] \times [-1.0, 1.0]
\end{equation}

These bounds match the TurtleBot3 Burger's physical velocity limits.

\subsection{Reward Function}
\label{sec:reward}

The reward guides the agent to reach the goal quickly and safely:

\begin{tcolorbox}[colback=lightgreen, colframe=accentgreen, title=\textbf{Reward Structure}]
\small
\[
R(s_t, a_t) = 
\begin{cases}
\textcolor{accentgreen}{\mathbf{+2000}} & \text{if distance to table} < 0.35\text{m} \\
\textcolor{accentred}{\mathbf{-500}} & \text{if collision} \\
50(D_{t-1} - D_t) - 0.5 - 0.3|\omega_t| & \text{otherwise}
\end{cases}
\]
\normalsize
Where the ``otherwise'' reward has three components:
\begin{itemize}
  \item \textcolor{primaryblue}{$50(D_{t-1} - D_t)$}: \textbf{Progress reward} (Dijkstra distance)
  \item \textcolor{warnorange}{$-0.5$}: \textbf{Time penalty} (per step)
  \item \textcolor{accentred}{$-0.3|\omega_t|$}: \textbf{Rotation penalty} (anti-zigzag)
\end{itemize}
\end{tcolorbox}

\begin{itemize}
  \item \textcolor{primaryblue}{\textbf{Progress reward}}: Uses the Dijkstra distance map (not Euclidean), so the robot is rewarded for navigating through corridors correctly.
  \item \textcolor{warnorange}{\textbf{Time penalty}} ($-0.5$ per step): Forces the robot to find short paths, prevents infinite loops.
  \item \textcolor{accentred}{\textbf{Rotation penalty}} ($-0.3 \cdot |\omega|$): Penalizes spinning, produces smooth trajectories.
\end{itemize}

\subsection{PPO Loss Function}

PPO clips the policy update to prevent catastrophic changes:

\begin{equation}
L^{\text{CLIP}}(\theta) = \mathbb{E}_t \left[ \min\left( r_t(\theta) \hat{A}_t,\ \text{clip}(r_t(\theta), 1 - \epsilon, 1 + \epsilon) \hat{A}_t \right) \right]
\end{equation}

where $r_t(\theta) = \frac{\pi_\theta(a_t | s_t)}{\pi_{\theta_{\text{old}}}(a_t | s_t)}$ is the probability ratio, $\hat{A}_t$ is the advantage, and $\epsilon = 0.2$.

Total loss:
\begin{equation}
L(\theta, \phi) = -L^{\text{CLIP}}(\theta) + 0.5 \cdot L^{\text{VF}}(\phi) - 0.01 \cdot H(\pi_\theta)
\end{equation}

\subsection{Analysis}

\subsubsection{Path Efficiency}
The RL agent produces \textbf{shorter paths} (36.38\,m vs.\ 42.50\,m), a \textbf{14.4\% improvement}. This is because the RL agent moves in \textbf{continuous space} (smooth curves and diagonals), while grid planners are limited to 4-connected cardinal movements.

\subsubsection{Success Rate}
Classical planners achieve 100\% (mathematically guaranteed). The RL agent achieves 100\% after sufficient training (2M steps with target KL stabilization).

\subsubsection{Obstacle Avoidance}
Classical planners guarantee $\geq 25$\,cm clearance via obstacle inflation. The RL agent comes as close as 7\,cm -- safe in simulation but risky for real deployment.

\subsubsection{Computation Time}
Classical planners are \textbf{$\sim$90$\times$ faster} at inference (15--60\,ms vs.\ 1.4\,s). However, the RL model requires no re-computation if the map changes -- only re-training offline.

\subsection{Potential Improvements}
\begin{itemize}
  \item \textbf{Curriculum learning}: Start with nearby goals, progressively increase difficulty
  \item \textbf{Safety layer}: Override RL actions when obstacle distance $< 15$ cm
  \item \textbf{Hybrid approach}: Use A* for global path + RL for local control (smooth following)
  \item \textbf{Domain randomization}: Add noise during training to improve sim-to-real transfer
\end{itemize}

\section{Conclusion}

Classical path planning (A* + DWA) provides a \textbf{reliable, fast, and safe} solution. Reinforcement learning (PPO Actor-Critic) produces \textbf{shorter, smoother paths} with continuous control, but requires training time and offers weaker safety guarantees.

For a production restaurant robot, a \textbf{hybrid approach} would be ideal: classical planning for guaranteed path safety combined with RL for smooth, natural robot motion.
